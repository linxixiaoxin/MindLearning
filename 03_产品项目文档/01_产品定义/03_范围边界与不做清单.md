# 范围边界与不做清单

> 创建时间：2026-04-29  
> 最近更新：2026-04-29  
> 当前状态：v0.1 / v0.2 边界稿  
> 关联产品：思想伙伴选配器

## 1. 范围判断

当前项目不是“完整知识平台重建”，而是一次内容方法论产品化验证。

第一阶段只回答一个问题：

> 用户能不能从一个真实困境出发，经过轻量选择，得到一个可信、可保存、可行动、可继续探索的思想伙伴组合？

## 2. v0.1 已进入范围

| 模块 | 范围 |
| --- | --- |
| 产品入口 | `/tools/thought-partner`、顶部导航、书库首页产品实验入口 |
| 困境输入 | 8 个典型困境，允许用户选择当前最接近的困境 |
| 能力轴 | 观看、内在、关系、社会、叙事、行动 |
| 轻量参数 | 承接方式、表达气质、当前目标 |
| 思想伙伴 | 主镜头、翻译者、校正者、行动伙伴、表达伙伴 |
| 结果页 | 困境重述、能力轴、学科镜头、伙伴组合、最小行动、继续探索路径 |
| 解释机制 | 展示“为什么这样匹配”的简要解释 |
| 保存图片 | Canvas 导出结果卡核心信息版 PNG |
| 行动后反馈 | 输入行动后回馈文本，生成洞察卡 |
| Codex 交接 | 复制 Prompt、粘贴 JSON、本地 inbox / response 半自动链路 |
| AI 接口占位 | Netlify Function 占位，接口不可用时回退本地规则 |

## 3. v0.2 候选范围

| 模块 | 说明 | 进入条件 |
| --- | --- | --- |
| 继续探索可点击 | 把结果里的专题页、书籍节点、N 卡从文本升级为链接 | v0.1 结果卡结构稳定 |
| 正式 AI 洞察 Prompt | 为行动后回馈生成更稳定洞察卡 | 收到足够真实回馈样本 |
| 场景库扩展 | 从 8 个困境扩展到 12-20 个困境 | 新困境来自真实反馈或已有 N 卡 |
| 推荐解释增强 | 每个伙伴补更明确的推荐理由和适用边界 | 用户反馈显示“可信度”仍不足 |
| 反馈收集入口 | 加静态表单、二维码或外部反馈链接 | 小红书承接路径开始测试 |
| 移动端分享优化 | 优化图片尺寸、文字换行和保存体验 | 浏览器测试确认主要问题 |

## 4. v1.0 以后再考虑

| 模块 | 暂缓原因 |
| --- | --- |
| 用户登录 | 当前验证不依赖身份系统，会增加复杂度 |
| 历史结果保存 | 先验证用户是否愿意保存图片和回访 |
| 个人长期思想地图 | 需要足够反馈样本和稳定字段后再做 |
| 完整 AI 追问 | 需要明确安全边界、成本、Prompt 和质量标准 |
| 后台配置系统 | 当前数据量可以用代码配置维护 |
| 付费深度报告 | 商业化前需要先验证免费工具价值 |
| 小程序版本 | 先用网页验证入口、传播和使用链路 |
| 大而全知识图谱 | 当前最缺产品叙事和用户入口，不是图谱规模 |

## 5. 明确不做

短期不做：

- 不把它做成“万能心理咨询工具”。
- 不给出诊断式、治疗式或权威判断式结论。
- 不承诺 AI 生成内容一定准确。
- 不把所有作者强行塞进每一次推荐。
- 不为了展示书多而牺牲结果可理解性。
- 不在没有反馈证据时扩展复杂账户、会员和支付系统。

## 6. 需求进入规则

新需求进入开发前，至少回答 4 个问题：

1. 它服务哪个验证项：被理解、可信、收藏、行动、继续探索？
2. 它是否能在 v0.1-v0.2 范围内低成本验证？
3. 它是否会显著增加维护成本或理解成本？
4. 它是否需要真实用户反馈作为前置条件？

## 7. 范围更新机制

- 每次新增功能前，先更新本文件的范围判断。
- 每次上线后，把实际进入范围的内容同步到 `../03_产品方案与需求/24_版本需求池与优先级.md`。
- 如果出现“想做但不确定”的需求，先进入 `v1.0 以后再考虑`，等反馈证据足够再拉回。
