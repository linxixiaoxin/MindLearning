# 版本需求池与优先级

> 创建时间：2026-04-29  
> 最近更新：2026-04-29  
> 当前状态：v0.1 到 v0.2 需求池  
> 关联产品：思想伙伴选配器

## 1. 用途

本文件用于管理思想伙伴选配器后续需求，避免需求散落在聊天、开发记录和灵感文档里。

需求进入本池后，再决定：

- 是否进入当前版本。
- 是否需要用户反馈作为前置条件。
- 是否属于产品、内容、研发、运营或测试工作。
- 是否应该推迟到 v1.0。

## 2. 优先级定义

| 优先级 | 定义 |
| --- | --- |
| P0 | 不做就无法完成当前验证或影响核心体验 |
| P1 | 能显著提高 v0.2 验证质量，但不阻塞 v0.1 |
| P2 | 有价值，但需要更多证据或资源 |
| Later | 暂不进入近期版本，保留观察 |

## 3. 状态定义

| 状态 | 说明 |
| --- | --- |
| Backlog | 已记录，未排期 |
| Ready | 信息足够，可进入开发/执行 |
| In Progress | 正在推进 |
| Blocked | 被依赖项阻塞 |
| Done | 已完成 |
| Dropped | 明确不做或取消 |

## 7. 当前推荐推进顺序

### 第一组：验证可用性

1. R-001：8 个困境自测。
2. R-002：桌面保存图片检查。
3. R-003：移动端保存图片检查。
4. R-005：fallback 检查。

### 第二组：收集真实反馈

1. R-004：收集 5 条真实体验反馈。
2. 把反馈写入 `../02_用户研究与反馈/12_用户反馈池.md`。
3. 判断 R-101 / R-102 / R-103 谁先进入 Ready。

### 第三组：确定 v0.2

当前最可能的 v0.2 主线：

`继续探索可点击 + 正式 AI 洞察 Prompt + 小样本反馈闭环`

但最终应以反馈池证据为准。

新增想法进入后的建议分流：

1. 误区诊断可作为 v0.2 的独立工具候选，优先级高于领导心智评估，因为 `mindset-traps` 已有 K/N 卡骨架。
2. 旅行专题页先按内容实验推进，不阻塞当前工具研发；本次厦门/泉州行程可作为首个真实样本。
3. 领导者心智评估先做题库和表达边界，等题目足够稳定后再考虑进入开发。

## 8. 需求拆分原则

新增需求必须能回答：

1. 它服务哪个验证假设？
2. 它影响哪个用户流程？
3. 它是否有验收标准？
4. 它是否需要新数据结构？
5. 它是否会扩大范围边界？

如果无法回答，先保留在 Later，不进入近期开发。

## 9. 每周更新规则

每次推进后更新：

- 新增需求。
- 状态变化。
- 优先级变化原因。
- 已完成需求的验收结果。
- 被反馈证据推翻或加强的需求。
