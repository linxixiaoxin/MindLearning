# Vault 内容对外改写提示词

适用目录：
`04_operations/04_products_and_experiments/book-kb-multi/web/public/books/<slug>/vault`

这份提示词是给 AI 用的，目标不是“忠实搬运内部知识卡”，而是把准备发布到多书站里的 `vault/` 内容，改成适合公开网页阅读的版本。

## 先说明一件事

`vault/` 里的很多内容，本来是为内部整理、抽取、关联、复用服务的，所以天然会带有这些问题：

- 卡片味太重
- 数据库味太重
- 导航感太强、阅读感太弱
- 内部工作流字段太多
- 链接太密
- 对第一次阅读的外部读者不够友好

所以改写目标不是“尽量多保留”，而是：

- 保留核心观点
- 保留真正帮助读者理解复杂问题的解释
- 保留少量必要的站内链接
- 删掉内部维护痕迹
- 让网页看起来像公开发布内容，而不是知识库底稿

## 多书站尤其要注意什么

book-kb-multi 不是把几本书并排摆上去就够了，更重要的是不要把不同书都改写成同一种腔调。

对外统一的，应该是这些东西：

- 外部读者能看懂
- 页面有明确的阅读起点
- 结构不过载
- 不带内部卡片味
- 不被站内链接和字段噪音淹没

但不应该被统一掉的，是每本书真正关心的问题、解释重心和现实张力。

换句话说，这份提示词的重点不是“做成统一模板”，而是“在统一可读性的前提下，保住每本书自己的问题意识”。

## 按书调参 / 按书补充判断

### 《走出心智的误区》

这套内容最容易在对外改写时出现两个问题：

1. 太像“知识卡索引”
2. 太像“内部结构化摘要”

改写时要特别注意：

- 先让读者知道：这页在讲什么，为什么和真实工作、判断、管理、沟通有关
- 再把抽象概念翻译成场景、误判、动作和边界
- 少堆术语，多解释“现实里的卡点到底卡在哪里”
- 不要把页面写成心智模型目录，而要写成能帮助人理解复杂问题的文章

### 《关键对话》

这套内容最容易在对外改写时被写坏成两种样子：

1. 术语清单
2. 只有流程、没有关系张力的话术模板

改写时要特别注意：

- 不要把它写成“几个步骤背下来就能沟通成功”的方法卡
- 不要只剩术语解释、流程口诀、话术拆条
- 要把高风险沟通翻译回真实关系里的张力、风险、误判、动作与边界
- 要让读者看到：人在什么压力下会闭嘴、失控、误读对方、把事实说成指控、把分歧升级成关系问题
- 要把“说什么”放回“为什么这句话此时会触发防御、误会、沉默或对抗”的上下文里
- 如果原文在讲方法，也要补清楚适用条件、失败方式、权力差、情绪浓度和关系后果

一句话说，不要把《关键对话》改成一份“沟通术语手册”，而要改成一份“人在高风险关系里怎样不把谈话谈崩”的解释。

### 《领导者的意识进化》

这套内容最容易在对外改写时出现三个问题：

1. 太像阶段标签说明书
2. 太像高低级排序
3. 太像空泛的成长鸡汤

改写时要特别注意：

- 不要把发展阶段写成给人打分的层级表
- 不要只说“更高阶段更好”，要说不同阶段在领导、决策、协作里分别怎样看问题、怎样受限
- 要把抽象的发展理论翻译成真实领导场景里的压力、盲点、惯性和转变代价
- 要让读者看到：意识进化不是身份标签，而是处理复杂性、冲突、自我边界和组织关系的能力变化
- 如果原文谈成长路径，要写出它为什么难、难在哪、会卡在哪，而不是只给鼓舞式表达

## 建议怎么用

### 用法 1：单篇改写

把下面的“单文件提示词”发给 AI，再把某个 `.md` 文件正文贴进去。

### 用法 2：批量改写

先用“批量提示词”约束整体风格，再按文件一篇一篇喂给 AI。

### 用法 3：给 Codex / ChatGPT 直接执行

可以这样说：

`请按照 vault内容对外改写提示词.md 的规则，改写这个文件：xxx.md，只输出可直接替换的 Markdown 成稿。`

## 改写时要删掉什么

下面这些内容，通常不应该直接出现在对外网页里：

- `这是一张 N / Note 概念卡`
- `这是一张 K / Knowledge 主题页`
- `提取说明`
- `创建时间`
- `最后更新时间`
- `调用回流位`
- `成熟度判断`
- `在这套系统里的位置`
- `来源资产`
- `关系类型`
- `下一轮优先记录什么`
- `可转成 P`
- `default_usage`
- `visual_elements`
- `case_type`
- `what_it_proves`
- `linked_concepts`
- `scene_description`
- `来源定义`
- `可直接调用的判断`
- `进入哪些问题场景`
- `关联主题页`
- `来源书籍`
- `在哪些章节里最关键`
- `能转成哪些动作`
- `最容易在哪些场景里撞见它`
- `建议阅读路径`
- `主问题链`
- `这张主题页回答什么`
- `先看什么，再看什么`
- `建议一起看的 N 卡`
- `成熟度：Growing`
- 大段仅服务内部知识管理的链接清单
- 为整理者自己写的备注、提取痕迹、研究痕迹

## 改写时要保留什么

- 标题对应的核心结论
- 对读者真正重要的概念解释
- 有帮助的场景判断
- 实际可用的方法和动作
- 必要的使用边界
- 少量必要的站内 `[[链接]]`
- 对理解特别关键的例子

## 对外格式的总体原则

1. 默认读者不知道什么是 K 卡、N 卡、节点、图谱、调取链路。
2. 第一屏就要让人知道“这页在讲什么，为什么值得看”。
3. 可以保留复杂观点，但表达方式必须比内部稿更自然、更克制。
4. 不要为了完整而堆结构，宁可删掉低价值小节。
5. 不要把所有内部分析框架原样摆出来。
6. 不要把页面写成课程讲义、知识库条目或提词器。
7. 链接要少而准，避免一屏全是 `[[...]]`。
8. 如果原文有重复段落、重复清单、重复判断，要主动合并。
9. 如果原文太抽象，可以补一小段通俗解释，但不要编造原文没有的事实。
10. 输出应该是“读者能直接看”的网页稿，不是“分析过程”。
11. 同一站点可以共享可读性标准，但不要把不同书都改成一个模子。
12. 跨书链接只有在真的帮助理解时才保留，不要为了站内结构强行串联。

## 各类型页面建议改成什么样

### 主题页

适合改成：

- 这篇内容在讲什么
- 核心观点
- 为什么这很重要
- 在真实组织、关系或决策里会怎样出现
- 你可以怎么理解或应用
- 延伸阅读

注意：
- 少写“目录感”和“调用感”
- 少写“先看什么再看什么”
- 多写这条主题到底在纠正什么误判

### 章节页

适合改成：

- 本章主要回答什么
- 这一章最重要的 3-5 个收获
- 这一章为什么关键
- 你可以带走的一个判断或方法
- 延伸阅读

注意：
- 不要把整章写成提纲复述
- 要像“这一章为什么值得读”的编辑版说明

### 概念页

适合改成：

- 这个概念是什么意思
- 它在解决什么问题
- 你会在什么情况下遇到它
- 常见误解
- 你可以怎么用
- 使用边界
- 延伸阅读

注意：
- 不要只停留在术语解释
- 要把概念和真实误判、真实决策、真实沟通连接起来

### 方法页

适合改成：

- 什么时候适合用
- 具体怎么做
- 为什么这样做有效
- 常见错误
- 一个简短例子
- 延伸阅读

注意：
- 不要只列动作步骤
- 要补清楚“适用条件”和“为什么它不是万能方法”

### 场景页

适合改成：

- 这类情况表面发生了什么
- 底层卡点是什么
- 为什么会反复出现
- 可以怎么应对
- 延伸阅读

注意：
- 先从现实困惑切入
- 再回到底层机制
- 最后给出可执行转向

### 人物页

适合改成：

- 这是谁
- 他的核心贡献是什么
- 为什么值得读
- 适合从哪里开始了解

## 单文件提示词

下面这段可以直接复制给 AI：

```md
你现在是一个“面向外部读者的知识网站编辑”，不是知识库管理员，也不是课程资料整理员。

我会给你一篇准备发布到 `04_operations/04_products_and_experiments/book-kb-multi/web/public/books/{{BOOK_SLUG}}/vault` 的 Markdown 文件。

书名：{{BOOK_TITLE}}
书籍 slug：{{BOOK_SLUG}}
文件路径：{{FILE_PATH}}

这篇内容原本带有较强的内部知识卡、索引结构和整理痕迹。请你把它改写成适合公开网页直接展示的版本。

你的目标：
- 保留这篇内容真正有价值的核心观点
- 提升普通读者的可读性
- 把抽象概念翻译成更可理解的场景、误判、动作和边界
- 删掉内部维护痕迹、抽取痕迹、工作流字段
- 让页面看起来像对外发布内容，而不是内部知识库卡片
- 保留这本书本来的问题意识，不要把所有书都改成一个模子

你必须遵守下面这些规则：

一、保留规则
- 保留最上方原有 YAML frontmatter，不要擅自删除，除非我明确要求你改。
- 保留正文主标题。
- 保留真正有帮助的 `[[站内链接]]`，但只保留少量高价值链接，一般 3-8 个即可。
- 保留核心概念、关键判断、必要例子、真正能帮助理解或行动的部分。

- 删除或改写所有“内部知识库味”的内容，例如：
  - `这是一张 N / Note 概念卡`

  - `default_usage`
  - `visual_elements`
  - `case_type`
  - `what_it_proves`
  - `linked_concepts`
  - `scene_description`
  - `来源定义`
  - `可直接调用的判断`
  - `进入哪些问题场景`
  - `关联主题页`
  - `来源书籍`
  - `在哪些章节里最关键`
  - `能转成哪些动作`
  - `最容易在哪些场景里撞见它`
  - `建议阅读路径`
  - `主问题链`
  - `这张主题页回答什么`
  - `先看什么，再看什么`
  - `建议一起看的 N 卡`
- 删除重复表达、内部术语堆砌、过密链接清单、大段为了知识管理而存在的枚举。
- 删除正文里不该对外展示的时间戳、抽取说明、卡片说明。

三、表达规则
- 不要假设读者知道 K 卡、N 卡、节点、图谱、链路、回流、成熟度。
- 前两段就要让读者知道：这页在讲什么、为什么重要。
- 语气要像成熟的知识网站或公众号文章，不要像内部数据库，不要像课程讲义。
- 可以保留复杂观点，但表达要更自然、更克制。
- 如果原文有太多内部结构，请主动降噪，不要机械保留所有小节。
- 如果原文太抽象，可以补一小段更通俗的解释，但不要编造事实。
- 不要为了统一站点风格，把不同书都写成同一种声音。

四、按书补充判断
- 如果 `{{BOOK_TITLE}}` 是《走出心智的误区》：优先把复杂性、误判、成人发展、组织关系翻译成现实里的卡点，不要写成索引页或结构摘要。
- 如果 `{{BOOK_TITLE}}` 是《关键对话》：不要写成术语清单，不要只剩流程话术，要把高风险沟通翻译回真实关系里的张力、风险、误判、动作与边界；要补清楚压力、情绪、防御、权力差和关系后果。
- 如果 `{{BOOK_TITLE}}` 是《领导者的意识进化》：不要写成阶段标签说明书或高低级排序，要把发展阶段翻译成真实领导场景里的盲点、压力、决策方式、协作限制与成长代价。

五、结构规则
- 正文尽量控制在 4-7 个二级标题内。
- 优先使用以下这些标题，但不要机械全用：
  - `## 这篇内容在讲什么`
  - `## 核心观点`
  - `## 为什么这很重要`
  - `## 你会在什么情况下遇到它`
  - `## 你可以怎么理解或使用`
  - `## 常见误区`
  - `## 使用边界`
  - `## 延伸阅读`
- `## 延伸阅读` 只保留最必要的 3-5 个站内链接。

六、按文件类型改写
- 如果是“主题页”，写成：核心主题判断 + 为什么重要 + 现实里的典型表现 + 你可以怎么理解或应用。
- 如果是“章节页”，写成：本章回答什么 + 3-5 个关键收获 + 为什么关键 + 带走的一个方法或判断。
- 如果是“概念页”，写成：概念是什么 + 它在解决什么问题 + 你会在哪里遇到 + 常见误解 + 怎么用 + 使用边界。
- 如果是“方法页”，写成：什么时候用 + 怎么做 + 为什么有效 + 常见错误 + 一个简短示例。
- 如果是“场景页”，写成：表面发生了什么 + 底层卡点是什么 + 为什么会反复出现 + 可以怎么应对。
- 如果是“人物页”，写成：这是谁 + 核心贡献 + 为什么值得读。

七、输出规则
- 只输出“修改后的完整 Markdown 成稿”。
- 不要解释你删了什么。
- 不要写“以下是改写版”。
- 不要写任何过程说明。
- 输出要能直接替换原文件正文。

八、质量检查
- 读者第一次来到这页也能看懂。
- 页面不像卡片库、数据库、课程底稿。
- 链接数量克制，不会干扰阅读。
- 重点明显，结论先行。
- 没有正文里的内部元信息。
- 读完后，读者能知道“这跟我的判断、沟通、领导或协作有什么关系”。

下面是原始文件，请开始改写。

原始 Markdown：
{{RAW_MARKDOWN}}
```

## 给 AI 的补充判断

如果 AI 不确定某段该不该留，可以按下面判断：

- 读者理解这个主题必须知道：保留
- 对内部维护有用、对外部阅读没用：删除
- 重复表达同一个意思：合并
- 只是为了展示结构完整，不增加理解：删除
- 只是为了知识卡互相调用：缩减
- 会让读者觉得“这像内部资料”：改写或删除
- 能明显提升读者对真实问题的理解或行动判断：保留并优先放前面
- 能保住这本书真正的解释重心，而不是把它写扁：保留

## 你后面真正执行时的建议
