## 1. 工程定位

`book-kb-multi` 当前是一个 Vite + Vue 的多书知识站前端项目，思想伙伴选配器作为其中第一个产品实验工具进入站内。

当前技术目标不是搭建完整平台，而是支撑：

- 多书站首页、书籍页、专题页和阅读器。
- 思想伙伴选配器交互工具。
- 静态配置驱动的推荐结果。
- 结果卡 / 洞察卡保存图片。
- 本地 Codex bridge 和 Netlify Function AI 接口占位。

## 2. 技术栈

| 层级 | 技术 / 文件 | 说明 |
| --- | --- | --- |
| 构建工具 | Vite | `web/package.json` |
| 前端框架 | Vue 3 | `web/src/main.js`、`web/src/App.vue` |
| 数据加载 | 静态 JSON + 前端模块 | `web/public/registry/books.json`、`web/src/data/thoughtPartnerData.js` |
| 图谱/可视化 | D3 | 现有知识图谱相关组件 |
| Markdown | marked | 阅读器解析 |
| 二维码 | qrcode | 站点现有依赖 |
| 本地桥接 | Node HTTP server | `web/scripts/codex-insight-bridge.mjs` |
| AI 接口 | Netlify Function | `web/netlify/functions/thought-partner-insight.mjs` |

## 4. 核心运行链路

### 4.1 多书站链路

```text
App.vue 路由判断
-> BookLibrary / TopicPage / ArticleReader 等组件
-> bookData.js 加载 registry、book、topic 数据
-> public/registry/books.json + public/books + public/topics
```

### 4.2 思想伙伴选配器链路

```text
App.vue
-> /tools/thought-partner
-> ThoughtPartnerTool.vue
-> thoughtPartnerData.js
-> 前端规则匹配
-> 结果卡 / 洞察卡
-> 保存图片 / AI 接口 / Codex bridge
```

## 5. 关键文件责任

| 文件 | 责任 |
| --- | --- |
| `web/src/App.vue` | 顶层页面状态、路由判断、导航入口 |
| `web/src/components/ThoughtPartnerTool.vue` | 思想伙伴选配器主交互、结果生成、保存图片、Codex 交接 |
| `web/src/data/thoughtPartnerData.js` | 能力轴、作者伙伴、困境、角色等核心配置 |
| `web/src/components/BookLibrary.vue` | 书库首页、产品实验入口、书籍/专题入口展示 |
| `web/src/lib/bookData.js` | registry、book、topic 数据加载与默认结构兜底 |
| `web/public/registry/books.json` | 书籍、专题、工具入口注册表 |
| `web/scripts/codex-insight-bridge.mjs` | 本地 Codex 收件箱/返回通道 |
| `web/netlify/functions/thought-partner-insight.mjs` | 线上 AI 洞察接口占位 |

## 6. 当前数据流

### 6.1 结果卡

```text
用户选择困境和参数
-> selectedScene / selectedNeed / selectedTaste / selectedGoal
-> partnerSet + 参数权重
-> partnerFor(role)
-> resultAxes / matchReason / adaptiveMinimumAction
-> share-card 展示
-> Canvas 绘制导出 PNG
```

### 6.2 洞察卡

```text
用户输入 actionFeedback
-> generateFeedbackInsight()
-> 优先尝试 Netlify Function
-> 失败时 fallback 本地规则
-> feedbackInsight 展示
-> 可保存图片
```

### 6.3 Codex 半自动链路

```text
页面生成结构化 payload
-> POST http://127.0.0.1:8787/thought-partner/inbox
-> 写入 web/codex-inbox/thought-partner-latest.json
-> Codex 读取并生成 response
-> 写入 thought-partner-response.json 或 POST /thought-partner/response
-> 页面检查返回并应用洞察卡
```

## 7. 本地开发命令

在 `web/` 下执行：

```text
npm run dev
npm run build
npm run preview
npm run codex:bridge
```

说明：

| 命令 | 用途 |
| --- | --- |
| `npm run dev` | 启动 Vite 开发服务器 |
| `npm run build` | 构建生产包 |
| `npm run preview` | 本地预览构建产物 |
| `npm run codex:bridge` | 启动本地 Codex bridge，默认端口 8787 |

## 8. 环境变量

| 变量 | 用途 | 默认 |
| --- | --- | --- |
| `CODEX_BRIDGE_PORT` | 本地 bridge 端口 | `8787` |
| `OPENAI_API_KEY` | Netlify Function 调用 AI 所需 key | 无 |
| `OPENAI_MODEL` | Netlify Function 使用模型 | 代码内当前默认值 |

## 9. 开发约束

- 优先维护静态配置和前端规则，避免过早引入后台系统。
- 新增困境、作者伙伴、能力轴时，同步更新 `31_数据结构与配置字典.md`。
- 新增接口或 bridge 字段时，同步更新 `32_API与本地桥接协议.md`。
- 新增功能需求时，同步更新 `36_研发任务与需求同步.md` 和 `../03_产品方案与需求/24_版本需求池与优先级.md`。
- 发现缺陷或技术债时，同步更新 `35_缺陷与技术债记录.md`。
