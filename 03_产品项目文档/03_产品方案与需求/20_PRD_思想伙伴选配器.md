# PRD：思想伙伴选配器

> 创建时间：2026-04-29  
> 最近更新：2026-04-29  
> 当前状态：v0.1 / v0.2 PRD 基线稿  
> 所属项目：`book-kb-multi`  
> 产品入口：`/tools/thought-partner`

## 1. Overview

### 1.1 Problem Statement

当前多书知识站已经有书籍、专题、搜索和阅读器，但用户仍然容易从“书很多、内容很深”感受到距离。对真实用户来说，最自然的入口往往不是“我想读哪本书”，而是：

- 我现在卡在一个说不清的困境里。
- 我想知道这件事可以从哪些角度理解。
- 我需要一个今天能试的小行动。

因此需要一个工具，把真实困境转译成思想能力轴、作者伙伴组合和最小行动，让知识站从“资料陈列”变成“问题承接”。

### 1.2 Solution Summary

思想伙伴选配器提供一个轻量交互流程：

`选择/描述当前困境 -> 选择承接方式、表达气质、当前目标 -> 生成思想伙伴组合 -> 保存结果卡 -> 行动后回填 -> 生成洞察卡`

第一版不追求完整 AI 平台，而是用轻量规则和半自动 Codex 交接验证产品方向。

### 1.3 Target Users

| 用户类型 | 典型需求 |
| --- | --- |
| 小红书内容读者 | 从一篇共鸣笔记进入更完整的理解路径 |
| 自我探索用户 | 把模糊困境转成可理解、可表达、可行动的结构 |
| 读书/知识用户 | 不只看书单，而是知道哪些书能帮自己看当前问题 |
| 创作者/产品观察者 | 看见一套内容方法论如何被产品化 |

## 2. Goals & Success Metrics

### 2.1 Goals

1. 验证真实困境入口是否成立。
2. 验证思想伙伴组合是否具备理解和收藏价值。
3. 验证行动后回馈和洞察卡是否能形成二次互动。
4. 为后续个人思想地图、AI 追问、专题页承接提供证据。

### 2.2 Success Metrics

| Metric | Current Baseline | Target | Timeline |
| --- | --- | --- | --- |
| 有效体验反馈 | 0-少量内部反馈 | 5-20 条有效反馈 | v0.1 测试期 |
| 被理解感正向反馈 | 未系统收集 | 至少 5 条 | v0.1 测试期 |
| 保存图片正向信号 | 未系统收集 | 至少 3 条 | v0.1 测试期 |
| 行动后回馈样本 | 未系统收集 | 至少 3 条 | v0.2 前 |
| 继续探索意愿 | 未系统收集 | 至少 3 条 | v0.2 前 |

### 2.3 Non-Goals

- 不做心理诊断、治疗建议或权威判断。
- 不做用户登录、会员、付费报告。
- 不做完整历史结果保存。
- 不做重型 AI 问答后台。
- 不把所有书、作者和概念强行纳入一次推荐。

## 4. Scope

### 4.1 In Scope

- 8 个典型困境。
- 6 个思想能力轴。
- 5 类思想伙伴角色。
- 15 个作者 / 思想 / 方法配置。
- 轻量问答控件：承接方式、表达气质、当前目标。
- 结果卡：困境重述、能力轴、学科镜头、伙伴组合、最小行动、推荐解释、继续探索。
- Canvas 保存图片。
- 行动后反馈输入。
- 洞察卡生成和保存图片。
- Codex 复制 Prompt / JSON 回填 / 本地 inbox / response 交接。
- Netlify Function AI 接口占位和本地规则 fallback。
- 思想伙伴作者主页：人物简介、系统方法论、适用问题域、核心概念、概念 / 方法 / 场景库、知识图谱雏形。

### 4.2 Out of Scope

- 用户登录。
- 历史结果中心。
- 付费报告。
- 小程序版本。
- 后台管理系统。
- 完整 AI 多轮对话。

### 4.3 Future Considerations

- 个人长期思想地图：需要多次反馈和用户意愿验证。
- 可点击继续探索路径：v0.2 优先候选。
- 正式 AI 洞察 Prompt：需要真实行动后回馈样本。
- 场景库扩展：需要来自用户反馈或现有 N/K 卡证据。
- 作者知识图谱：先用作者主页承接本地书籍和 K/N 卡，条目足够后再升级为可探索图谱。

## 5. Solution Design

### 5.1 Functional Requirements

#### 困境与参数选择

- FR-1：系统必须展示 8 个典型困境，并允许用户选择其一。
- FR-2：系统必须展示承接方式、表达气质、当前目标等参数。
- FR-3：参数变化必须影响伙伴角色权重或解释标签。

#### 结果生成

- FR-4：系统必须生成困境重述。
- FR-5：系统必须显示本次能力轴和说明。
- FR-6：系统必须生成五类思想伙伴组合。
- FR-7：系统必须展示“为什么这样匹配”的解释。
- FR-8：系统必须给出最小行动。
- FR-9：系统必须展示继续探索路径。

#### 保存与分享

- FR-10：用户必须能把结果卡保存为 PNG。
- FR-11：保存失败时必须显示可理解的失败提示。
- FR-12：保存图片应优先保证中文换行、核心信息完整和移动端可读性。

#### 行动后洞察

- FR-13：用户必须能输入行动后反馈文本。
- FR-14：系统必须能生成洞察卡，包含模式、说明、新建议和标签。
- FR-15：AI 接口不可用时必须回退到本地规则版。

#### Codex 交接

- FR-16：系统必须支持复制结构化 Prompt。
- FR-17：系统必须支持粘贴 Codex JSON 并应用成洞察卡。
- FR-18：系统必须支持本地 inbox / response 文件交接链路。

#### 作者主页与概念库

- FR-19：结果卡中的作者名字必须可点击，并进入该作者的长期方法主页。
- FR-20：作者主页必须说明该思想伙伴的系统方法论、适用问题域、典型提问方式和使用提醒。
- FR-21：作者主页必须展示概念 / 方法 / 场景库，用于承接本地书籍、章节卡、K 卡和 N 卡搬运来的结构化内容。
- FR-22：作者主页必须保留人物知识图谱雏形入口，后续可连接概念、书籍、场景、方法和案例。
- FR-23：结果卡里的轻量展开说明只解释“当前困境下他怎么帮你想”，不替代作者主页。

### 5.2 User Experience

核心体验应当轻而清楚：

1. 用户先看到困境，而不是看到书单。
2. 用户通过少量参数表达“现在想被怎样承接”。
3. 结果页把推荐解释成一组功能位，而不是权威答案。
4. 保存图片服务于回看和分享，不追求花哨。
5. 行动后反馈用于二次理解，不制造压力。

### 5.3 Edge Cases

| Scenario | Expected Behavior |
| --- | --- |
| 用户没有写行动后反馈 | 提示先写一段反馈再生成洞察卡 |
| AI 接口不可用 | 自动回退本地规则洞察 |
| Codex JSON 格式错误 | 显示解析失败提示，不覆盖现有结果 |
| 图片导出失败 | 显示失败原因或通用失败提示 |
| 参数组合导致候选过少 | 使用默认权重补足伙伴组合 |
| 移动端宽度较窄 | 文案换行，按钮和卡片不重叠 |

## 6. Technical Considerations

### 6.1 Constraints

- 当前为 Vue 前端项目。
- 当前匹配规则主要在前端数据与组件中维护。
- AI 接口为 Netlify Function 占位，需配置 `OPENAI_API_KEY` 后才能真实调用。
- 本地 Codex bridge 依赖本机服务和 `web/codex-inbox/` 文件。

### 6.2 Integration Points

- `web/src/data/thoughtPartnerData.js`：困境、能力轴、伙伴和规则数据。
- `web/src/components/ThoughtPartnerTool.vue`：主要交互页面。
- `web/netlify/functions/thought-partner-insight.mjs`：AI 洞察接口占位。
- `web/codex-inbox/thought-partner-latest.json`：本地请求文件。
- `web/codex-inbox/thought-partner-response.json`：本地返回文件。

### 6.3 Data Requirements

当前不应收集敏感身份信息。用户反馈如需归档，应脱敏保存，避免记录真实姓名、联系方式和可识别隐私。

## 7. Dependencies & Risks

### 7.1 Dependencies

| Dependency | Owner | Status | Impact if Delayed |
| --- | --- | --- | --- |
| 真实反馈样本 | 运营/产品 | 待收集 | 无法判断 v0.2 优先级 |
| 浏览器导出测试 | 研发/质检 | 待执行 | 分享卡体验不稳定 |
| AI Prompt 样本 | 内容/方法论 | 待补 | 洞察卡质量不稳定 |
| 继续探索链接映射 | 内容/研发 | 待补 | 站内深访链路弱 |

### 7.2 Risks

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| 用户觉得推荐太抽象 | M | H | 强化推荐理由和最小行动 |
| 结果像鸡汤或诊断 | M | H | 保持克制表达，避免权威判断 |
| 图片导出移动端不稳 | M | M | 建立测试用例和降级提示 |
| AI 输出不可控 | M | H | fallback 本地规则，限制输出字段 |
| 需求膨胀 | H | M | 用范围边界文档控制 |
