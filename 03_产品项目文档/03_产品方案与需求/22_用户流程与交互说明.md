# 用户流程与交互说明

> 创建时间：2026-04-29  
> 最近更新：2026-04-29  
> 当前状态：v0.1 交互基线稿  
> 关联页面：`/tools/thought-partner`

## 1. 总流程

```text
进入工具
-> 选择当前困境
-> 调整承接方式 / 表达气质 / 当前目标
-> 查看思想伙伴结果
-> 保存结果卡
-> 尝试最小行动
-> 输入行动后回馈
-> 生成洞察卡
-> 保存洞察卡 / 继续探索专题或书籍节点
```

## 2. 入口流程

| 入口 | 用户预期 | 页面行为 |
| --- | --- | --- |
| 顶部导航“思想伙伴” | 直接进入工具 | 打开 `/tools/thought-partner` |
| 书库首页“产品实验” | 从多书站了解这是一个工具 | 打开工具页 |
| 小红书内容引导 | 从具体困境进入 | v0.2 可考虑预选困境 |
| 专题页引导 | 从问题地图进入工具 | v0.2 可考虑带入专题上下文 |

## 3. 选择困境

第一屏应优先让用户看到“这是不是我现在的问题”，而不是先看作者和书。

当前 8 个困境：

- 刷完小红书更讨厌自己的生活
- 恋爱里总觉得自己不够被选择
- 明明没人逼，却总觉得自己不够努力
- 房间很整洁，但总觉得不像自己的家
- 聚会后很累，不知道自己是不是太装
- 信息太多，越看新闻越焦虑
- 总觉得自己的人生卡在旧剧本里
- 懂了很多道理，但生活没有任何变化

交互要求：

- 当前选中困境要有明确状态。
- 困境不应写得像诊断，应像可进入的生活场景。
- 后续可增加“我想写自己的困境”输入，但 v0.1 可先不做。

## 4. 轻量问答控件

### 4.1 承接方式

作用：表达用户现在想被怎样帮助。

可能影响：

- “被安放”提高翻译者、表达伙伴权重。
- “被推动”提高行动伙伴权重。
- “被校准”提高校正者权重。

### 4.2 表达气质

作用：影响结果的语言和作者候选倾向。

可能影响：

- “诗性”偏向巴什拉、亚当·菲利普斯等。
- “行动”偏向 L 先生、福格、非暴力沟通等。
- “结构”偏向系统思考、管理和组织相关作者。

### 4.3 当前目标

作用：明确本次结果是为了理解、表达、行动还是继续探索。

可能影响：

- “找到行动”强化行动伙伴和最小行动。
- “形成表达”强化表达伙伴。
- “理解自己”强化内在和观看轴。

## 5. 结果卡

结果卡是核心产品产物，应服务“保存、回看、分享”。

结果卡应包含：

| 模块 | 说明 |
| --- | --- |
| 困境重述 | 用更清楚但不评判的语言重述用户困境 |
| 本次能力轴 | 展示系统如何理解这个困境 |
| 学科母题镜头 | 给出更大的问题视角 |
| 思想伙伴组合 | 五类角色及对应作者/方法 |
| 为什么这样匹配 | 解释推荐逻辑 |
| 最小行动 | 给用户一个今天能尝试的小动作 |
| 继续探索 | 推荐专题页、书籍节点或 N 卡 |

交互要求：

- 结果不应像“标准答案”，而应像“陪你从几个角度看”。
- 五类伙伴的功能位要清晰，避免作者列表化。
- 最小行动必须足够小，避免变成新的压力。

## 6. 保存图片

保存图片用于验证结果卡是否有收藏和传播价值。

要求：

- 点击后显示生成中状态。
- 成功后导出 PNG。
- 失败后提示用户重试或改用截图。
- 导出内容优先保留核心信息，不要求完整复刻 DOM。
- 中文换行、标题、隐喻关系图和底部正文不得重叠或裁切。

## 7. 行动后回馈

用户完成或尝试最小行动后，可以输入一段反馈。

输入引导应鼓励具体描述：

- 我实际做了什么？
- 当时有什么感受？
- 我注意到自己有什么模式？
- 哪里卡住了？

不要让用户觉得必须写得很完整。短文本也应可用。

## 8. 洞察卡

洞察卡用于把行动后反馈转成二次理解。

当前字段：

- 我看见的模式
- 这说明什么
- 新的建议
- 标签

交互要求：

- 如果 AI 接口不可用，回退本地规则版。
- 如果用户粘贴 Codex JSON，解析成功后应用到洞察卡。
- 洞察卡也应支持保存图片。

## 9. Codex 交接流程

### 9.1 复制 Prompt

用户点击复制后，页面生成结构化 Prompt，用户可粘贴到当前 Codex 对话。

### 9.2 JSON 回填

Codex 返回固定结构 JSON 后，用户粘贴回页面，页面解析并应用为洞察卡。

### 9.3 本地收件箱

页面发送请求到：

```text
http://127.0.0.1:8787/thought-partner/inbox
```

写入：

```text
web/codex-inbox/thought-partner-latest.json
```

Codex 写回：

```text
web/codex-inbox/thought-partner-response.json
```

页面通过：

```text
http://127.0.0.1:8787/thought-partner/response
```

检查返回并应用洞察卡。

## 11. 作者主页与概念库流程

### 11.1 入口关系

结果卡里的思想伙伴分两层：

- 卡片内的轻量展开：解释这个人在当前困境里怎么帮你想、适合用在什么场景、可以从哪个问题开始。
- 作者名字点击：进入作者主页，展示这个人的长期方法论、核心概念库和后续知识图谱入口。

作者主页不绑定某一个具体困境。结果卡只是一次调用，作者主页负责沉淀“这个人长期能提供什么思考工具”。

### 11.2 作者主页内容

作者主页至少包含：

| 模块 | 作用 |
| --- | --- |
| 人物简介 | 说明这个思想伙伴为什么进入系统 |
| 系统方法论 | 解释他通常如何分析问题 |
| 适用问题域 | 帮用户判断什么时候适合找他 |
| 典型提问方式 | 给用户一个可直接带走的问题 |
| 核心概念 | 快速看到他的常用思想工具 |
| 概念 / 方法 / 场景库 | 承接本地书籍、K 卡、N 卡搬运来的结构化内容 |
| 人物知识图谱雏形 | 后续连接概念、书籍、场景、方法和案例 |

### 11.3 本地书籍与卡片接入原则

后续新增思想伙伴时，不建议只由前端硬写人物介绍。更好的路径是：

1. 优先导入或整理这个作者的本地书籍、章节卡、K 卡和 N 卡。
2. 把卡片抽成作者页里的概念、方法、场景、母题和来源条目。
3. 结果卡只调用其中一小部分，作者页保留完整入口。
4. 等条目足够多后，再升级为类似书库的作者知识图谱。

这样作者页既能作为当前最小入口，也不会变成孤立的百科页面。
