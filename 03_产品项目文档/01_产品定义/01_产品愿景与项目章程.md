# 产品愿景与项目章程

> 创建时间：2026-04-29  
> 最近更新：2026-04-29

> 关联项目：`book-kb-multi` / `思想伙伴选配器`

## 1. 项目一句话

把多书知识站从“展示很多书的资料库”，推进成一个能帮助用户从真实困境进入知识、得到思想伙伴组合、形成最小行动的认知地图产品。

## 2. 产品愿景

`book-kb-multi` 的长期愿景不是成为一个更大的书库，而是成为一个“认知地图产品”的公开样板间：

- 用户从真实生活困境进入，而不是从书名、概念或章节进入。
- 书、作者、学科和方法作为证据层与工具层出现。
- 最终产物不是知识罗列，而是可理解、可表达、可行动的个人认知地图。

`思想伙伴选配器` 是这个愿景下的第一个产品实验。

## 3. 为什么现在做

当前项目已经具备三个基础：

1. 多书站已经接入书籍、专题页、搜索、阅读器和 registry。
2. 账号方向已经从单篇内容表达，转向“用认知地图看见人、关系和知识的运转方式”。
3. 思想伙伴选配器已经完成 v0.1 前端能力：困境输入、能力轴、伙伴组合、结果卡、洞察卡、保存图片、Codex 交接。

现在最需要补的不是资产数量，而是产品闭环：

`小红书触发 -> 网站工具承接 -> 结果保存/反馈 -> 反哺内容和产品迭代`

## 4. 项目目标

### 用户目标

- 让用户觉得自己的困境被准确重述。
- 让用户理解为什么某些作者、书和方法适合当前问题。
- 让用户得到一个今天可以尝试的最小行动。
- 让用户愿意保存、截图、分享或继续探索相关专题。

### 产品目标

- 验证“从真实困境进入知识站”是否比“从书进入知识站”更有效。
- 验证“思想伙伴组合”是否能成为多书内容的产品化表达方式。
- 验证结果卡、洞察卡、继续探索路径是否能形成可复用的产品组件。

### 账号与内容目标

- 承接小红书内容带来的信任和共鸣。
- 把用户反馈沉淀成场景库、N/K 卡、专题页和后续笔记选题。
- 展示“内容方法论产品化”的能力，而不仅是写作能力。

## 5. 项目范围

当前阶段聚焦 `思想伙伴选配器`，不把整个多书站一次性重做。

本阶段核心对象：

- `/tools/thought-partner`
- 书库首页的产品实验入口
- 顶部导航入口
- 结果卡与洞察卡
- 本地 Codex 交接链路
- 后续与专题页 / 书籍节点 / N 卡的连接

## 6. 关键成功标准

| 成功维度 | v0.1-v0.2 观察信号 |
| --- | --- |
| 被理解感 | 用户认为困境重述准确，愿意继续读结果 |
| 推荐可信度 | 用户能理解“为什么是这些思想伙伴” |
| 收藏价值 | 用户愿意保存图片、截图或转发 |
| 行动价值 | 用户愿意尝试最小行动，并回填行动后反馈 |
| 探索价值 | 用户愿意点开相关专题、书籍节点或 N 卡 |
| 产品方向 | 用户表达想要长期思想地图、更多困境或个人化版本 |

## 7. 项目角色

| 角色 | 责任 |
| --- | --- |
| 产品负责人 | 定义目标、范围、优先级、验证标准 |
| 内容/方法论负责人 | 维护困境、能力轴、作者伙伴、结果解释和表达边界 |
| 研发负责人 | 维护前端、数据结构、桥接协议、构建和发布 |
| 运营负责人 | 设计小红书承接、反馈收集、用户语料归档 |
| 质检负责人 | 检查可用性、文案可信度、图片导出、移动端体验 |

当前阶段这些角色可以由同一协作体承担，但文档上要保持分工清晰。

## 8. 当前项目原则

- 先验证真实用户感受，不先追求完整平台。
- 先用轻量规则验证方向，不先构建重型 AI 后台。
- 先把结果做成可保存、可分享、可反馈的卡片，不只做页面展示。
- 先把 5-20 条真实反馈跑通，再扩展更多功能。
- 所有新需求必须能回到一个验证问题：它是否让用户更容易被理解、行动、保存或继续探索。
