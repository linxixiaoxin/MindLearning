## 一、立项定位

`思想伙伴选配器` 是多书知识站里的第一个产品实验工具。

它不作为一本书进入书库，也不作为普通跨书专题页存在，而是作为一个可交互工具：

`用户当前困境 -> 思想能力轴 -> 学科母题镜头 -> 作者伙伴组合 -> 最小行动`

第一版目标不是做完整平台，而是验证：

- 用户是否愿意从真实困境进入知识站。
- 作者、书和学科能否以“思想伙伴组合”的方式被理解。
- 结果页是否有收藏、转发和继续追问的价值。

## 三、当前 MVP 范围

### 已进入 v0.1 的能力

- 8 个典型困境。
- 6 个思想能力轴：观看、内在、关系、社会、叙事、行动。
- 5 类思想伙伴角色：主镜头、翻译者、校正者、行动伙伴、表达伙伴。
- 15 个作者 / 思想 / 方法配置。
- 轻量问答控件：承接方式、表达气质、当前目标。
- 结果页：困境重述、能力轴、学科镜头、伙伴组合、最小行动、继续探索路径。

### 暂不进入 v0.1

- 用户登录。
- 历史结果保存。
- AI 追问。
- 后台配置系统。
- 付费深度报告。
- 小程序版本。

## 六、下一步开发顺序

1. 跑通本地构建与浏览器检查。
2. 优化结果页可截图效果。
3. 给每个结果补 1 条“为什么这样匹配”的更完整解释。
4. 把继续探索路径接到已有专题页、书籍节点或 N 卡。
5. 收集 5-20 个真实用户反馈。
6. 再判断是否进入 AI 追问和个人思想地图。

## 七、验证标准

第一轮不是看访问量，而是看用户是否真的觉得结果有用。

| 验证项 | 达标信号 |
| --- | --- |
| 被理解感 | 用户觉得困境重述准确 |
| 推荐可信度 | 用户能理解为什么推荐这些伙伴 |
| 收藏价值 | 用户愿意截图或保存结果 |
| 继续探索 | 用户愿意点开相关卡片、书或专题 |
| 产品方向 | 用户主动想继续生成“我的长期思想地图” |

## 八、当前立项结论

这个项目正式进入 `book-kb-multi` 产品实验线。

第一阶段项目定义为：

`内容方法论产品化验证项目`

而不是：

`大型知识平台 / 完整小程序 / 会员产品`

一句话：

`先让思想坐标系变成一个真实用户能点、能选、能保存、能继续追问的工具。`

## 九、2026-04-28 最小产品化补充

### 本轮新增

- 结果页升级为可保存的分享卡片。
- 新增“保存图片”按钮，当前通过前端把结果卡片 DOM 转成 PNG 下载。
- 补充“为什么这样匹配”的解释，让作者组合不只是罗列名单。
- 新增“行动后回馈”输入区。
- 新增“行动回馈洞察卡”，包含：
  - 我看见的模式
  - 这说明什么
  - 新的建议
  - 标签
- 洞察卡同样支持保存图片。
- 新增 Codex 交接模式：
  - 一键复制结构化 Prompt 给当前 Codex 对话。
  - 支持把 Codex 返回的 JSON 粘贴回页面并应用成洞察卡。
- 新增本地 Codex 收件箱模式：
  - 启动命令：`npm run codex:bridge`
  - 本地接口：`http://127.0.0.1:8787/thought-partner/inbox`
  - 最新请求文件：`web/codex-inbox/thought-partner-latest.json`
  - 页面点击“发送给本地 Codex”后，Codex 可直接读取这个 JSON 文件继续分析。
- 新增本地 Codex 返回通道：
  - 返回接口：`http://127.0.0.1:8787/thought-partner/response`
  - 返回文件：`web/codex-inbox/thought-partner-response.json`
  - 页面支持“检查 Codex 返回”，读取后自动应用成洞察卡。
  - 页面发送请求后会短时间轮询返回接口，若 Codex 已写回结果则自动应用。
- 轻量问答控件升级为真实匹配参数：
  - `承接方式` 会影响五类伙伴角色权重，例如“被安放”提高翻译者 / 表达伙伴权重，“被推动”提高行动伙伴权重。
  - `表达气质` 会影响作者候选权重，例如“诗性”偏向巴什拉 / 亚当·菲利普斯，“行动”偏向 L 先生 / 福格 / 非暴力沟通。
  - `当前目标` 会影响角色权重和最小行动收束，例如“找到行动”强化行动伙伴，“形成表达”强化表达伙伴。
  - 结果卡会显示本次调参标签，让用户知道当前组合是按哪些选择生成的。
- `本次能力轴` 增加说明文案：
  - 观看 / 社会 / 内在等轴不是用户要再选择的问题，而是系统对当前困境的理解维度。
  - 这些轴会影响作者伙伴匹配权重和结果解释角度。
- `保存图片` 改为更稳的 Canvas 绘制导出：
  - 不再依赖浏览器对 DOM foreignObject 截图的支持。
  - 点击后会显示生成状态或失败提示。
  - 当前导出的是结果卡核心信息版 PNG，优先保证可用性。
  - 2026-04-29 修正导出画布高度和中文换行，避免生成顶部有字、下方大面积空白的长图。
  - 2026-04-29 提升导出像素密度，并把导出文案从内部字段改为用户用途说明。
- 新增隐喻关系图：
  - 结果卡中以“当前困境”为中心，五类思想伙伴围绕它形成不同功能位。
  - 主镜头负责照亮入口，翻译者负责把雾气翻译成人话，校正者负责校准方向，行动伙伴负责铺踏脚石，表达伙伴负责给它一种说法。
  - 保存图片时同步绘制一张简化关系图，让结果不只是文字清单。
  - 2026-04-29 修正导出图中标题与隐喻节点重叠、底部正文被裁切的问题。
- 新增思想伙伴理解入口：
  - 每个伙伴补充 `thinkingLogic / useFor / starterQuestion`。
  - 结果页保留卡片内轻量三条点击展开：他怎么帮你想、适合用在、起手问题，避免结果页过长。
  - 卡片内三条属于“这个作者在当前问题里的调用方式”，放在该伙伴卡的核心概念下方。
  - 点击作者名字进入独立思想伙伴详情页，不再额外保留“进入主页”按钮。
  - 详情页路由：`/tools/thought-partner/thinkers/:thinkerId`。
  - 详情页展示该伙伴的介绍、适合轴线、系统方法论、适用问题域、典型提问方式、核心概念、适合位置和使用提醒。
  - 作者主页是长期方法库，不绑定单个困境；结果卡只是一次调用。
  - 详情页新增“核心概念库”图谱雏形，后续可升级为人物知识图谱，连接概念、书籍、场景和方法卡。
  - 保存图片中的伙伴组合也会带上起手问题，避免只展示陌生作者名。
- 后续扩展策略：
  - 轻量补充人物或方法，可以先由 Codex 按现有字段直接加到 `thoughtPartnerData.js`，适合验证产品手感。
  - 如果要把某位作者作为核心长期伙伴，最好走书籍或资料导入，提取其关键概念、论证方式、适用场景和误用风险。
  - 人物数量扩大后，需要把作者配置从单个 JS 文件拆成可维护的数据表或 JSON 配置。
- 新增 Netlify Function 接口占位：
  - `web/netlify/functions/thought-partner-insight.mjs`
  - 后续配置 `OPENAI_API_KEY` 后可直接走真实 AI 分析。

### 当前实现边界

- 当前网页不能直接调用这次聊天里的 Codex 实例。
- 已支持“复制给 Codex -> Codex 生成 JSON -> 回填页面”的人工交接链路。
- 已支持“发送给本地 Codex -> 写入本地收件箱 -> Codex 读取文件 -> 写回返回文件 -> 页面应用”的半自动交接链路。
- 页面点击“生成洞察卡”时会优先尝试调用 AI 接口；接口不可用时才退回本地规则版。
- 当前匹配规则仍是轻量启发式评分，不是机器学习模型；它用于验证用户是否能感知到“选择会改变结果”。
- 真实 AI 调用输入为：
  - 用户原始困境
  - 所选能力轴
  - 思想伙伴组合
  - 最小行动
  - 用户行动后回馈文本

### 下一步建议

1. 真实浏览器里测试 PNG 保存效果，重点看中文、卡片宽度和移动端截图。

4. 再收 5-20 条真实回馈文本，决定洞察卡字段是否稳定。
